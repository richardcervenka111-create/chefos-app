01 The pass

Four findings, graded by Sautero's own system

Kitchen Flow has four states: done, in progress, waiting, blocked. The same four states describe the project itself.

DoneStation: Engineering

The defence layer is at real SaaS standard

The deploy gate runs the full lock suite against the candidate build on localhost before production sees anything. Auditors block commits, pushes and deploys alike. Every entry in the incident log carries a root cause and a permanent guard. This is rare even in teams of ten.

169 migrations · 49 tables · 100% under RLS
19 UI invariants · 3 blocking auditors · rollback ~3 min
In progressStation: Architecture

One file carries the entire product

app/index.html is 24,802 lines and 1.27 MB — 17,714 of those lines are JavaScript inside a single <script> block holding 979 functions. Nothing can be imported from it, so the unit tests have to cut functions out of the source with a hand-written brace-matching scanner. That is a lot of extra work caused by one decision.

Architecture 75/100 — the weakest block in the report

02 Diagnosis of today's two hours

The problem isn't that function. It's time-to-truth.

There is no JS runtime on the machine — neither Playwright nor node runs locally. So every behavioural check has to travel through GitHub Actions: install Playwright, download Chromium, sign in the QA accounts, run the whole lock suite. And it runs whole, even for a one-line change.

Today
~8–12 min per attempt
Local, single tile
~30 s per attempt
Across eight iterations that is the difference between ~80 minutes of waiting and ~4 minutes. Two hours on one function is not a lapse in focus — it is exactly what comes out when the feedback loop is measured in minutes instead of seconds.

05 How billion-dollar SaaS companies build this

Six practices — three already here, three missing

The interesting part isn't what those companies have on top. It's that three of the things built here solo are usually only introduced after a company's first serious outage.

PracticeHow they do itSautero
Test pyramid Roughly 70% fast unit tests (milliseconds), 20% integration, 10% browser E2E. E2E is the most expensive and most brittle layer, so it gets the smallest share. Inverted pyramid: almost everything is an E2E lock suite, unit tests are a handful.missing
Local dev loop Vite or esbuild, hot reload, tests firing on file save. Nobody at Stripe waits on CI to find out whether a button works. With no local JS runtime, every check is a round trip through CI.missing
Test selection Bazel, Nx, Turborepo — the diff determines which tests are affected and only those run. The rest runs overnight. The full suite runs on every change, including a one-line CSS edit.missing
Feature flags Trunk-based development: everything merges to main, unfinished work hides behind a switch (LaunchDarkly, Statsig). No long-lived branches. featureEnabled() plus internal QA accounts — exactly that model.have it
Blameless postmortems After an incident you write the root cause and the permanent fix, not the culprit. The Google SRE book made this the industry standard. HEALTH_LOG.md: date, cause, guard. Own mistakes included.have it
Deploy gate + rollback Nothing ships without green tests; returning to the previous version is one click and a few minutes. Lock gate against the candidate build plus a ~3 min rollback workflow.have it

06 Order of moves

Four steps that change the trajectory

Ranked by hours released or risk closed per hour invested. Not by what is interesting to build.

01

Get a local runtime

Node 22, pip install playwright and chromium on the machine where the code is written. Then http.server 8080 and tile_lock_test.py --only recipes locally. This is the one step that directly fixes today's two hours.

~30 minone-off
returns hours weekly
02

Tier the CI by speed

Cache the Playwright browsers (actions/cache on ~/.cache/ms-playwright). On push, run the static auditors plus the suite for the tile being touched; keep the full lock suite for merges to main and a nightly run. The gate stays — it just stops charging for every line.

~2 hone-off
−60% CI time
03

Cut the first module into a .js file

Not a rewrite. One module — ideally a calculation one (cost, hours) — moved out as an ES module with export, loaded via <script type="module">. Still buildless. The payoff: unit tests without a scanner that carves functions out of source, and a first diff that can actually be read.

~1 dayper module
a repeatable pattern
04

Freeze the product for two weeks

No new tiles. Einzelfirma and AHV, an invoice, a payment path, and German across at least the twenty most-used screens. These four things stand between a finished product and a first invoice — and they are the only items on the list that writing better code cannot make faster.

~10 dayswith no new code
unlocks CHF 490/year
